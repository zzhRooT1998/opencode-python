## 2. 背景
- 现有命令行开发流程中，开发者需要频繁在终端、编辑器、测试输出和代码搜索之间切换，完成一次“小任务”往往要经过理解代码、定位文件、执行命令、修改验证、回顾结果多个环节。
- 通用聊天模型可以回答代码问题，但缺少本地工具执行、权限控制、上下文检索和会话追踪，难以稳定完成真实代码任务。
- `opencode-python` 的目标是在本地 CLI 中提供一个可控、可解释、可恢复的代码助手，优先覆盖个人开发者的日常 coding workflow。

## 3. 产品定位
### 3.1 产品定义
- 一个面向个人开发者的本地 CLI coding assistant。
- 以 Python 后端实现，不引入 TypeScript 后端依赖。
- 首版采用多 Agent 协作模式，围绕计划、执行、复核形成任务闭环。

### 3.2 目标用户
- 日常在本地终端中开发、调试、重构和写测试的 Python 开发者。
- 需要一个比纯聊天更强、但比完整 IDE Agent 更轻量的本地工具的开发者。

### 3.3 核心价值
- 减少代码定位和上下文切换成本。
- 提供可控的工具执行与权限防护。
- 让任务过程可追踪、结果可解释、异常可恢复。

## 4. 目标与非目标
### 4.1 产品目标
- 支持用户通过 CLI 输入一个代码任务，系统自动完成理解、计划、执行、审查和回复。
- 在代码任务中集成轻量 RAG，降低模型缺乏仓库上下文导致的误判和误改。
- 对工具执行提供统一的权限控制和审计能力。
- 支持会话持久化，便于查看历史任务和恢复上下文。

### 4.2 非目标
- 不追求完整复刻上游 TUI 或全部交互体验。
- 不做云端同步、多人协作、权限共享或远端托管。
- 不做语音、多模态、网页端 UI。
- 不做全语言通用后端，MVP 只保证 Python 后端实现。
- 不把 Symbol Graph 作为 MVP 或 v1 的正式承诺能力。

## 5. 核心使用场景
### 5.1 修复 bug
- 用户描述一个报错、失败测试或异常行为。
- 系统检索相关代码与日志上下文，制定排查步骤，执行只读命令或修复动作，并给出验证结果。

### 5.2 增加小功能
- 用户描述一个范围明确的小功能。
- 系统理解目标代码、生成实施步骤、修改相关文件、运行验证命令，并输出变更摘要。

### 5.3 代码阅读与定位
- 用户询问某个模块、函数或功能入口位置。
- 系统通过检索和代码阅读返回引用路径、行号和关键调用关系。

### 5.4 补测试或回归验证
- 用户要求为已有逻辑补充测试，或对变更进行验证。
- 系统执行测试命令，收集结果，并根据失败信息继续修正或中止。

## 6. MVP 范围
### 6.1 CLI 能力
- 提供 `chat` 命令用于交互式任务。
- 提供 `run` 命令用于一次性任务执行。
- 提供 `session list` 和 `session show` 用于查看会话历史和关键信息。

### 6.2 多 Agent 任务闭环
- `Planner`：根据用户目标、历史上下文和检索摘要生成结构化步骤计划。
- `Executor`：在权限约束下调用模型和工具完成当前步骤，内部采用 ReAct-style 执行循环。
- `Reviewer`：根据结果质量、约束和风险决定接受、重试或中止。
- 用户侧保持单入口体验，不要求显式选择 Agent。

### 6.3 轻量 RAG
- 文件发现与分块：按文件、类、函数或适合阅读的段落进行切片。
- 召回：优先使用关键词与 BM25 召回相关片段。
- rerank：对候选片段进行重排，提高与任务的相关性。
- 上下文注入：为 Agent 提供带来源路径与行号的 evidence pack。

### 6.4 工具与安全
- `shell`：执行命令，要求有超时、输出截断和权限控制。
- `fs_read`：读取文件或范围内容。
- `fs_write`：在允许路径下写入文件。
- `search`：对仓库执行文本搜索，优先利用 `rg`。
- 权限模式支持 `allow / ask / deny`，默认 `ask`。

### 6.5 会话与输出
- 保存会话、消息、关键事件和任务产物。
- 输出最终结论、执行摘要、关键来源、失败原因和必要的下一步建议。
- 支持在非交互环境中使用 CLI 输出。

## 7. 用户体验要求
- 用户只需要输入自然语言任务，不需要手动串联检索、执行和审查流程。
- 需要引用代码时，输出必须包含来源文件路径与行号。
- 遇到高风险命令时，系统必须显式阻断或要求确认，不能静默执行。
- 当任务失败时，系统必须说明失败发生在哪个步骤，以及是权限、工具、模型还是验证失败导致。

## 8. 非功能需求
### 8.1 可靠性
- 单任务支持状态持久化与恢复。
- 单步失败时支持有限次重试，避免无限循环。

### 8.2 安全性
- 危险命令默认不放行。
- 所有工具调用都必须留下审计信息。

### 8.3 性能
- 中等规模仓库下，单任务中位耗时目标不超过 60 秒。
- 检索与索引更新不应阻塞整体交互到不可用。

### 8.4 可解释性
- 检索结果、工具执行和审查结论应能被追踪。
- 输出应尽量说明采用了哪些代码证据和执行结果。

## 9. 成功指标
- 任务成功率（人工评估）>= 70%
- 危险命令误放行率 = 0
- 单任务中位耗时 <= 60s（中等仓库）
- 用户可以在历史会话中回溯一次任务的核心事件和结果

## 10. 版本边界
### 10.1 MVP 必做
- CLI 入口
- 多 Agent 执行主链路
- 轻量 RAG
- 工具调用与权限控制
- 会话持久化与恢复
- OpenAI-compatible provider

### 10.2 后续增强
- embedding 混合检索
- 面向任务类型的检索策略模板
- 基于 Python AST 的轻量符号索引评估
- 更丰富的回归评测和任务模板库

### 10.3 明确延后
- Symbol Graph 主线化
- 多语言通用语义索引
- 云端协作和团队权限系统

## 11. 验收标准
- 用户可以通过 `chat` 或 `run` 发起真实代码任务，并获得完整任务结果。
- 系统在任务执行过程中能完成检索、计划、工具执行和复核，而不是只生成文本回答。
- 输出中能展示至少一个代码来源引用和明确的执行结果摘要。
- 高风险命令不会被默认静默执行。
- 会话可以被保存并通过 CLI 查询。

## 12. 风险与约束
- 多 Agent 会增加时延和成本，需要通过限制步骤数和重试次数控制。
- 轻量 RAG 召回质量不足会直接影响执行效果，需要保留来源信息和 rerank 机制。
- 工具权限策略过松会带来安全风险，过严会降低可用性，MVP 以安全优先。
